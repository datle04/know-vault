# Claude Code Prompts for KnowVault

This document contains carefully designed prompts for initiating Claude Code on the KnowVault project. Use the right prompt for the right context.

---

## Bilingual convention (read first)

This project uses **English for technical artifacts** and **Vietnamese for conversation**. This split is intentional:

**English (always):**
- Code (variables, functions, comments)
- Commit messages, branch names
- ADRs, PRD references, technical documentation
- README, file paths
- Technical terms even when speaking Vietnamese ("repository pattern", "vector embedding")

**Vietnamese (in conversation):**
- Explanations from Claude Code to you
- Proactive reminders (git, skills.md, etc.)
- Proposals and trade-off discussions
- Clarifying questions
- Recommendations and decisions

**Why this split:**
- English code maintains portfolio defensibility for international audience
- Vietnamese conversation reduces cognitive load — you process and decide faster in your native language
- Technical English terms stay English to avoid awkward translation and aid Google-ability

**The prompts below are written in English for two reasons:**
1. Claude Code parses English instructions more reliably for technical work
2. You can use these prompts verbatim by copy-pasting

**But Claude Code's RESPONSE should be in Vietnamese** — this is enforced by CLAUDE.md Rule #4 and explicitly stated in Prompt #1.

If Claude Code starts responding in English, gently redirect:

```
Per CLAUDE.md Rule #4, vui lòng respond bằng tiếng Việt. 
Code và commit messages giữ tiếng Anh, conversation tiếng Việt.
```

After the first 1-2 sessions, Claude Code should remember the convention.

## Prompt 1: Initial setup (first time only)

**When to use:** The very first time you start Claude Code on this project, after the repo is created with all documentation files committed.

**What it does:** Forces Claude Code to read ALL documentation, internalize the project philosophy, confirm understanding, and commit to the working patterns before writing any code.

**The prompt:**

```
Chào Claude. Đây là session đầu tiên trên project KnowVault.

Trước khi làm BẤT KỲ việc gì, tôi cần bạn fully internalize philosophy 
và conventions của project này. Hãy:

1. Đọc các file sau theo đúng thứ tự:
   - VISION.md (lý do WHY — product intent và success criteria)
   - PRD.md (cái gì WHAT — technical specifications và architecture)
   - CLAUDE.md (như thế nào HOW — working conventions và protocols)
   - README.md (public-facing context)
   - docs/skills.md (evidence mapping cho portfolio)

2. Sau khi đọc, tóm tắt lại cho tôi bằng tiếng Việt trong 5-7 bullet points:
   - Điều quan trọng nhất về project này
   - "Success" nghĩa là gì ở đây (và KHÔNG có nghĩa là gì)
   - Multi-Perspective Exploration methodology
   - Các proactive reminder protocols (git workflow + skills tracking)
   - Stack decisions đã locked và KHÔNG được re-litigate
   - Domain layer phải structure như thế nào

3. Confirm bạn hiểu các principles quan trọng:
   - Đây là LEARNING-DRIVEN project, không phải product-driven
   - External users là bonus, không phải success criteria
   - Tests viết từ ngày đầu (không có "no tests" phase)
   - Multi-Perspective Exploration nghĩa là implement nhiều approaches 
     GENUINELY, không phải "bad code rồi good code"
   - Bạn sẽ PROACTIVELY remind tôi về git discipline và skills.md updates
   - Khi không chắc, hỏi trước khi deviate từ PRD spec

4. Hỏi tôi 3-5 clarifying questions về bất cứ gì còn ambiguous hoặc 
   cần confirm trước khi bắt đầu Phase 0.

**Quan trọng về ngôn ngữ:** Per CLAUDE.md Rule #4, hãy giao tiếp với 
tôi bằng tiếng Việt cho mọi conversation, explanation, reminder. Giữ 
English cho code, commit messages, branch names, file paths, technical 
documentation (ADRs, PRD references, etc.). Technical terms như 
"repository pattern", "vector embedding", "TypeScript" giữ nguyên 
English ngay cả trong câu tiếng Việt.

KHÔNG bắt đầu implementation work. KHÔNG đề xuất alternative stacks. 
KHÔNG suggest skipping bất kỳ documentation reading nào.

Sau khi chúng ta aligned về các điều trên, tôi sẽ ra hiệu để bạn bắt 
đầu Phase 0.
```

### Why this prompt works

- **Explicit reading order:** VISION before PRD before CLAUDE forces Claude Code to absorb the WHY before the HOW
- **Summary requirement:** Tests genuine comprehension, not just "I read it"
- **Confirmation of principles:** Catches misunderstandings BEFORE they become code
- **Clarifying questions:** Surfaces ambiguities you didn't realize were ambiguous
- **Explicit "do not":** Prevents common Claude Code failure modes (rushing to code, proposing alternatives)
- **Final gate:** "Wait for green light" maintains your control

### Expected response shape

Claude Code should respond with:
1. Confirmation that all 5 files were read
2. The 5-7 bullet summary (substantial, not generic)
3. Explicit acknowledgment of the 6 critical principles
4. 3-5 thoughtful clarifying questions
5. Statement that they're awaiting your green light

If response is shorter than expected, missing items, or jumps to code — push back: "You haven't completed the framework. Please re-read CLAUDE.md and provide the full response."

## Prompt 2: Phase 0 kickoff

**When to use:** After Claude Code has read all docs and confirmed understanding (Prompt 1 above).

**The prompt:**

```
Great, you're aligned. Let's start Phase 0 (Project Setup).

Per PRD Section 12, Phase 0 includes:
1. Initialize pnpm monorepo with workspaces
2. Set up Next.js, NestJS, Prisma, Neon
3. Configure ESLint, Prettier, Husky
4. Set up GitHub Actions skeleton
5. Create initial README, CLAUDE.md (already done)
6. Set up Sentry, basic logging
7. Initialize ADR-0001 documenting initial stack choices

Before you write any code:

1. Check current branch and working tree status (per CLAUDE.md git protocol)
2. Propose how we should split Phase 0 into atomic feature branches
   (per CLAUDE.md commit size discipline)
3. Recommend which sub-task to tackle first and why

Wait for my approval on the branch structure before starting.
```

### Why this prompt works

- **References PRD section explicitly:** Forces Claude Code to consult, not improvise
- **Activates git protocol immediately:** First session establishes the discipline pattern
- **Asks for branch planning:** Tests if Claude Code internalized atomic commits
- **Wait for approval gate:** Keeps you in control of structure decisions

## Prompt 3: Daily session start (use every day)

**When to use:** Start of every coding session after Phase 0.

**The prompt:**

```
Starting a new session. Quick status check before we begin:

1. What's the current branch? Should we be on it or create a new one?
2. What's the working tree status? Any uncommitted work to address?
3. What was the last meaningful change? Where did we leave off?
4. Are there any skills.md updates pending from previous sessions?
5. What's the next task per PRD Phase [X]?

Based on the above, recommend what we should tackle this session and 
in what order. Wait for my confirmation before starting work.
```

### Why this prompt works

- **Activates the "start of session" protocol from CLAUDE.md:** Forces consistency
- **5 questions cover the full discipline:** git + skills + context + planning
- **Recommend + wait:** Maintains your decision authority

## Prompt 4: Mid-session exploration kickoff

**When to use:** When starting a Multi-Perspective Exploration (one of the 7 explorations).

**The prompt:**

```
We're starting Exploration [N] ([Topic]) per PRD Section 7 and CLAUDE.md.

This is a Multi-Perspective Exploration. Before any code:

1. Confirm: which Approach (A/B/C) are we implementing first?
2. Reference: what does PRD say about this Approach specifically?
3. Plan: what's the implementation scope to ship Approach [X] genuinely 
   (not minimal, not over-engineered)?
4. Measurement: what data do we need to collect to compare against 
   future approaches? Where will we store benchmarks?
5. Documentation: what ADR will we write at end of this Approach?

After we ship Approach [X], we'll measure, document, then move to the next.

Don't optimize for "preparing for Approach B" while implementing A. 
Build A as if it's the final answer. We compare AFTER, not DURING.
```

### Why this prompt works

- **Reinforces "genuine implementation":** Prevents Claude Code from cutting corners on Approach A
- **Forces measurement planning:** No benchmarks = no comparison = exploration wasted
- **ADR upfront:** Documentation is built-in, not afterthought
- **Anti-anticipation guidance:** Common failure mode where developers under-build A because "we'll redo it for B anyway"

## Prompt 5: End-of-phase audit (use at every phase boundary)

**When to use:** When transitioning between phases (e.g., Phase 1 → Phase 2).

**The prompt:**

```
Phase [X] is complete (or nearly so). Before moving to Phase [X+1], 
let's do a mandatory audit per CLAUDE.md protocols:

1. Skills.md audit:
   - Which skills can now be marked ✅ Evidenced based on this phase's work?
   - Which need 🟡 Partially evidenced?
   - For each, what specific code paths, ADRs, tests are the evidence?
   - Propose specific updates to docs/skills.md
   - Add entry to Update log

2. ADR audit:
   - Which significant decisions were made this phase?
   - Are all of them documented as ADRs?
   - Any decisions made but not yet ADR'd?

3. Phase deliverable check (per PRD Section 12):
   - What's done vs planned for this phase?
   - Anything intentionally deferred? Document why.

4. Postmortem candidates:
   - Any bugs, incidents, or wrong decisions worth postmortem?
   - Anything that surprised you (good or bad)?

5. README status update:
   - Update the "Project status" table in README.md

Propose all updates explicitly. Wait for my approval before applying.
```

### Why this prompt works

- **Forces the audit discipline:** End of phase is highest-leverage moment to update docs
- **5 sub-audits cover all the discipline dimensions:** skills, ADRs, deliverables, postmortems, README
- **Propose + wait:** Same pattern as other prompts

## Prompt 6: When Claude Code drifts (recovery)

**When to use:** When you notice Claude Code skipping the discipline (proposing alternatives, skipping branches, forgetting to remind, etc.)

**The prompt:**

```
Pause. I'm noticing pattern drift from CLAUDE.md protocols.

Specifically, [describe what you noticed — e.g., "you proposed switching 
to Drizzle ORM" or "you committed to main without prompting me" or 
"we've completed 3 features without skills.md updates"].

Per CLAUDE.md, the expected behavior is [reference the specific section].

Two things I need from you:
1. Acknowledge the drift without defensiveness
2. Re-read the relevant CLAUDE.md section and tell me what triggered 
   you to deviate

Then we'll calibrate: either CLAUDE.md needs adjustment (genuine reason 
to deviate) or you need to be more strict (you missed an enforcement moment).
```

### Why this prompt works

- **Specific accusation, not vague:** Claude Code can address the actual drift
- **References CLAUDE.md section:** Brings the protocol back into context
- **Two-part request:** Forces self-reflection (this often catches root cause)
- **Calibration framing:** Treats CLAUDE.md as living document, not stone tablets

## Prompt 7: Before opening a PR

**When to use:** When a feature branch is ready to merge.

**The prompt:**

```
Ready to open PR for branch [branch-name]. Per CLAUDE.md PR discipline:

1. Show me the full diff summary (files changed, line counts)
2. Show me the commit history on this branch
3. Are commits atomic per CLAUDE.md? Any that should be squashed?
4. Generate proposed PR description following CLAUDE.md template:
   - Title (conventional commit format)
   - ## What
   - ## Why  
   - ## How
   - ## Testing
   - ## Related (ADRs, PRD sections, issues)
5. List any skills.md updates this PR should trigger
6. Confirm all CI checks pass (typecheck, lint, test)

Wait for my approval on the PR description before opening.
```

### Why this prompt works

- **Comprehensive pre-PR audit:** Catches issues before they're public
- **Diff summary forces review:** Sometimes you realize the PR is too big
- **Skills.md trigger:** Connects the discipline loops
- **CI confirmation:** Don't open broken PRs

## Prompt 8: When stuck (Claude Code blocked)

**When to use:** When Claude Code says it's blocked or unclear on how to proceed.

**The prompt:**

```
You said you're blocked. Let's diagnose:

1. What's the specific decision or question you can't resolve?
2. Where in the documentation hierarchy did you look for guidance?
   - VISION.md (intent)
   - PRD.md (spec)
   - CLAUDE.md (patterns)
3. What did you find vs what was missing?
4. What are 2-3 possible paths forward, with trade-offs?
5. What would you recommend, and why?

If documentation has a gap, propose what should be added to which file 
to prevent this block in the future. If it's a genuine open question, 
let's discuss the trade-offs.

Don't sit on the block — surface it explicitly with context.
```

### Why this prompt works

- **Structured diagnosis:** Treats blocks as data, not failures
- **Documentation gap detection:** Continuous improvement of CLAUDE.md
- **Recommendation request:** Claude Code often has good instincts, but defaults to passive when blocked

## Anti-patterns to avoid

These are prompts that LOOK helpful but undermine the project:

### ❌ Don't say:
"Just build the article save endpoint quickly, we'll polish later."

**Why bad:** Triggers Claude Code's velocity bias. Contradicts CLAUDE.md "quality over speed."

**Say instead:**
"Implement the article save endpoint per PRD Section 5.5. Follow the full discipline — domain layer first, tests included, ADRs for any significant decisions."

---

### ❌ Don't say:
"What do you think we should do here?"

**Why bad:** Vague. Claude Code will hedge or propose multiple options without commitment.

**Say instead:**
"Per PRD Section X, the planned approach is Y. Are there reasons to deviate? If not, proceed with Y. If yes, state the reasons and propose alternative with trade-offs."

---

### ❌ Don't say:
"Skip the tests for this, it's a small change."

**Why bad:** Breaks "tests from day one" rule. Sets bad precedent.

**Say instead:**
"This is a small change. What's the minimum test that would catch a regression here? Add that, even if it's just one assertion."

---

### ❌ Don't say:
"Let's just commit everything to main this time."

**Why bad:** Breaks git workflow. Erodes habit.

**Say instead:**
"This needs to go on a branch. Suggest a branch name per CLAUDE.md convention."

---

### ❌ Don't say:
"You don't need to remind me about skills.md, I'll update it later."

**Why bad:** "Later" never comes. This is the failure mode skills.md was designed to prevent.

**Say instead:**
"Note the skills.md update needed. Add it as the next item in the current branch — we update before merging."

## Calibration after first 2 weeks

After roughly 2 weeks of using these prompts, do a calibration session:

```
We've been working on KnowVault for 2 weeks. Let's calibrate:

1. Which prompts have been working well?
2. Which prompts feel too rigid or generate friction?
3. Are there patterns I'm doing that aren't covered by any prompt?
4. Are there CLAUDE.md sections that need refinement based on actual practice?
5. Should we modify any prompts in this file?

Propose adjustments. We'll update accordingly.
```

This treats the prompt set as a living document, just like CLAUDE.md.

## Prompt 9: Explain mode (when you don't understand something)

**When to use:** When Claude Code mentions a concept, pattern, or term you don't fully understand. Use this instead of just nodding through.

**The prompt:**

```
Pause. Tôi không hiểu sâu về [concept/pattern/term].

Explain với tôi như mentor, bằng tiếng Việt, follow template:

1. **Vấn đề cốt lõi:** [concept] giải quyết vấn đề gì? 
   Tại sao nó tồn tại?

2. **Analogy:** Connect concept này với cái tôi đã biết 
   (Frontend, React, hoặc real-life examples)

3. **Trong KnowVault context:** [concept] apply như thế nào 
   ở project này? Code paths cụ thể?

4. **Trade-offs:** Khi nào dùng [concept]? Khi nào KHÔNG nên dùng?

5. **Common mistakes:** Junior devs hay confuse [concept] với 
   cái gì? Hay implement sai như thế nào?

6. **Recommended deeper learning:**
   - Quick read (30 phút): [resource]
   - Medium dive (vài giờ): [resource]
   - Deep mastery (vài tuần): [resource]

7. **Check my understanding:** Hỏi tôi 2-3 questions để 
   verify tôi hiểu đúng.

Sau khi tôi trả lời check questions, correct hoặc deepen tùy response.
```

### Why this prompt works

- **Explicit template** forces comprehensive explanation, không phải brief mention
- **Analogy requirement** ensures connection với existing knowledge
- **Trade-offs** prevent dogmatic thinking ("this is always best")
- **Common mistakes** is gold — junior dev pitfalls explicitly surfaced
- **Resources tiered by depth** lets you choose engagement level
- **Check understanding loop** forces active learning, không passive reading

### Example usage

```
Bạn: "Pause. Tôi không hiểu sâu về Hexagonal Architecture.
Explain với tôi như mentor, bằng tiếng Việt, follow template..."

Claude Code: [Full template response with analogy, trade-offs, 
common mistakes, resources, check questions]

Bạn: [Trả lời check questions]

Claude Code: [Correct/deepen based on understanding]
```

Don't skip this prompt when confused. "Just nodding through" is the #1 failure mode in learning-driven projects.

## Prompt 10: Phase-end learning review (mandatory at phase boundary)

**When to use:** At end of every phase, BEFORE moving to next phase. Pair with Prompt #5 (End-of-phase audit) — but this one focuses on learning, not just deliverables.

**The prompt:**

```
Phase [X] đã complete. Trước khi move sang Phase [X+1], conduct 
mandatory learning review per CLAUDE.md mentor protocol.

Review theo structure:

1. **Concepts fully covered trong phase này:**
   - List specific patterns, techniques, frameworks đã teach
   - Cho mỗi cái: brief recap (2-3 câu)

2. **Concepts touched on (chưa deep dive):**
   - List things mentioned briefly but chưa explore sâu
   - Recommend khi nào sẽ dig deeper

3. **Moments tôi struggled:**
   - Recall conversations where tôi seemed confused
   - Did we resolve? Or still gap?

4. **Self-assessment questions:**
   Hỏi tôi 3-5 questions check học sâu thật sự. Examples:
   - "Explain Repository pattern cho colleague trong 5 phút"
   - "Khi nào KHÔNG dùng [concept]?"
   - "Trade-offs giữa Approach A vs B trong Exploration X?"

5. **Reading recommendations cho phase tiếp:**
   - Based on next phase's concepts
   - Based on what tôi struggled với

6. **Skill consolidation:**
   - Update docs/skills.md với evidence từ phase này
   - Move ⏳ → 🟡 → ✅ where appropriate

Tôi sẽ trả lời self-assessment questions. Correct hoặc deepen 
dựa trên answers. Sau đó suggest reading priority.

Đây là consolidation moment — đảm bảo learning stick trước khi 
move on.
```

### Why this prompt works

- **Mandatory at phase boundaries** — prevents "ship and forget" mode
- **5 sub-reviews** cover all dimensions: covered, touched, struggled, tested, planned
- **Self-assessment** forces metacognition (thinking about thinking)
- **Skills.md integration** connects learning to evidence
- **Reading recommendations** plan next phase intentionally

### Expected duration

20-40 phút mỗi phase boundary. Đây là investment vào retention. Worth it.

### Pattern after multiple phases

Sau 3-4 phases, bạn build profile của:
- Concepts you've mastered (can teach others)
- Concepts you're shaky on (need revisit)
- Reading list tích lũy
- Skills evidence growing

Đây là portfolio gold — recruiter thấy structured learning approach, không phải haphazard.
